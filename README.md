# Tortilla Demand Forecasting

Machine learning project developed for a small tortilla distribution business to predict daily net demand for 
each client. The goal is to reduce product waste caused by overproduction and support daily production planning 
using historical data instead of manual estimation.

## Stack

`Python` · `Pandas` · `NumPy` · `Scikit-learn` · `XGBoost` · `LightGBM` · `CatBoost` · `RapidFuzz` · `Streamlit` · `Plotly`

## Problem

Production decisions were mainly based on experience, which frequently resulted in excess production, product returns, 
and financial losses.

The business had five years of daily sales records stored in Excel files with three different formats. In addition, 
client names were often written differently due to nicknames, spelling mistakes, or abbreviations, making the data 
difficult to analyze.



**Objective:** 
- Develop a complete pipeline to clean and process the historical data.
- Train a machine learning model to predict daily net demand.
- Provide the predictions through an interactive Streamlit dashboard with useful insights to support daily production decisions.

## Pipeline

`Raw Data` &rarr; `Cleaning` &rarr; `Name Standardization` &rarr; `Feature Engineering` &rarr; `ML Evaluation` &rarr; `Deployment`

1. **Raw Historical Data** — 3 Excel formats, 5 years
2. **Data Cleaning Pipeline** — 3 independent preprocessing modules
3. **Client Name Standardization** — RapidFuzz fuzzy matching
4. **Feature Engineering** — temporal variables, seasonal features, categorical encoding
5. **ML Evaluation** — XGBoost, LightGBM, CatBoost, Random Forest
6. **Streamlit Dashboard** — demand estimator + historical analytics

## Model Results
 
5-fold cross-validation on 80/20 train/test split. Target: `demanda_neta` (kg).
 
| Model         | MAE | RMSE | R²   |
|---------------|-----|------|------|
| Random Forest | 1.6896   | 2.5735    | 0.7358    |
| XGBoost       | 1.9784   | 2.8407    | 0.6780    |
| LightGBM      | 1.9870   | 2.8529    | 0.6753    |
| CatBoost      | 1.7935   | 2.6492    | 0.7200    |

**Random Forest** achieved the best performance.
> **Note:** The trained model is not included in this repository because its file size exceeds GitHub's limit. It can be generated by running `test_models_comparison.ipynb`.

## Notable Technical Decisions

**Fuzzy name matching:** Client names in the original records were often written differently because of nicknames, abbreviations, and spelling mistakes. A name matching module was developed using RapidFuzz to identify these variations and assign them to a single client, improving data consistency before training the models.

**Three cleaning notebooks:** The historical records were stored in three different Excel formats over five years. Instead of trying to process all of them with a single script, a separate notebook was created for each format. The cleaned data was then combined into a single dataset for the next stages of the pipeline.

```

```
